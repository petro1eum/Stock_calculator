# Отчет анализа оптимизатора запасов на основе Black-Scholes

## Резюме

Проект реализует оптимизацию запасов как задачу реального опциона: закупка товара рассматривается как колл-опцион на будущую выручку. После последних исправлений базовая финансовая интерпретация стала согласованной: ставка дисконтирования `r` применяется внутри Black-Scholes, а страйк `K` больше не содержит предварительно начисленные проценты.

**Текущая оценка: 8.7/10.** Математическая база стала чище, но модель все еще требует аккуратной калибровки волатильности выручки и проверки на реальных исторических данных.

## Текущая математическая интерпретация

### Параметры Black-Scholes

- `S` = ожидаемая будущая выручка от продаж при выбранном объеме запаса.
- `K` = стоимость исполнения складского решения: закупка плюс хранение.
- `T` = горизонт решения в годах: `weeks / 52`.
- `sigma` = волатильность будущей выручки.
- `r` = ставка дисконтирования.

Ключевой момент: `K` не должен одновременно включать начисление процентов и дисконтироваться внутри Black-Scholes. Поэтому корректная складская версия страйка сейчас выглядит так:

```typescript
export const calculateInventoryStrike = (
  quantity: number,
  purchase: number,
  hold: number,
  weeks: number
): number => {
  return quantity * purchase + quantity * hold * weeks;
};
```

Ставка капитала учитывается в самой формуле Black-Scholes через `K * exp(-r * T)`.

## Что исправлено

### 1. Детерминированный предел Black-Scholes

Раньше при почти нулевой волатильности функция возвращала внутреннюю стоимость `max(0, S - K)`. Это неверно при `r > 0`, потому что страйк должен дисконтироваться.

Актуальная логика:

```typescript
if (T <= 0) return { optionValue: Math.max(0, S - K) };
if (S <= 1e-6) return { optionValue: 0 };
if (K <= 1e-6) return { optionValue: S };
if (sigma <= 1e-6) {
  return { optionValue: Math.max(0, S - K * Math.exp(-r * T)) };
}
```

Такая же защита синхронизирована в C++/WASM реализации `blackScholesCall_cpp`.

### 2. Убран двойной учет ставки

Старая формула страйка была:

```typescript
const K = q * purchase * (1 + r * weeks / 52) + q * hold * weeks;
```

Проблема: проценты добавлялись в `K`, а затем `K` снова дисконтировался внутри Black-Scholes. Это смешивало будущую и текущую стоимость.

Новая формула:

```typescript
const K = calculateInventoryStrike(q, purchase, hold, weeks);
```

Она используется в основном калькуляторе, анализе товара, сценариях и строгой сценарной оценке.

### 3. Централизован расчет складского страйка

Вместо повторения формулы в разных компонентах добавлена единая функция `calculateInventoryStrike`. Это снижает риск расхождения математики между вкладками.

### 4. Добавлена годовая шкала для волатильности выручки

Если волатильность выводится из распределения выручки за горизонт `T`, ее нужно перевести в годовую шкалу перед передачей в Black-Scholes:

```typescript
export const annualizeLognormalVolatilityFromMoments = (
  mean: number,
  standardDeviation: number,
  years: number
): number => {
  if (mean <= 0 || standardDeviation <= 0 || years <= 0) return 0;
  const horizonSigma = Math.sqrt(Math.log(1 + (standardDeviation / mean) ** 2));
  return horizonSigma / Math.sqrt(years);
};
```

Эта логика используется в сценарной оценке `strictBSMixtureOptionValue` и синхронизирована с WASM-путем.

## Что уже реализовано корректно

### Ожидаемая выручка

Закрытая форма для ожидаемых потерянных продаж при нормальном спросе реализована корректно:

```typescript
const expectedLost = Math.max(
  0,
  demandStd * phi_z + (expectedDemand - q) * (1 - Phi_z)
);
const expectedSales = Math.max(0, expectedDemand - expectedLost);
```

Это соответствует формуле `E[(D - q)+]` для нормального спроса. Для более сложных случаев есть путь через Monte Carlo.

### Rush-поставки

Модель учитывает частичное покрытие дефицита экстренной закупкой:

```typescript
const rushSales = expectedLost * rushProb;
return expectedSales * fullPrice + rushSales * rushUnitRevenue;
```

Это корректная бизнес-эвристика: часть потерянного спроса может быть восстановлена, но с меньшей маржинальностью.

### Сценарная Black-Scholes оценка

`strictBSMixtureOptionValue` использует смесь сценариев и оценивает моменты выручки через симуляцию:

```typescript
const sigmaBS = annualizeLognormalVolatilityFromMoments(muRev, sigmaRev, T) || 0.2;
```

Это более строгий путь, чем простая эвристическая волатильность, потому что `sigma` строится из распределения будущей выручки.

## Оставшиеся модельные риски

### 1. Black-Scholes применим, но как модель реального опциона

Интуитивно складской запас действительно похож на опцион: бизнес платит `K`, чтобы получить право на будущую выручку `S`. Поэтому подход Black-Scholes здесь уместен.

Ограничение в том, что классический Black-Scholes предполагает торгуемый базовый актив, логнормальную динамику и непрерывное хеджирование. В запасах этого нет. Поэтому корректнее говорить не "биржевой Black-Scholes один в один", а "модель реального опциона с Black-Scholes оценкой".

### 2. Главная точка калибровки: `sigma`

Самый чувствительный параметр модели — волатильность будущей выручки. Текущая `calculateVolatility` смешивает спрос, валюту и логистику:

```typescript
const combined = Math.sqrt(
  base * base + currencyVol * currencyVol + logisticsVol * logisticsVol
);
```

Это разумная эвристика, но ее нужно калибровать на исторических данных. Для production-решений предпочтительнее чаще использовать `strictBSMixtureOptionValue`, где волатильность считается из симулированной выручки.

### 3. Распределение спроса

Текущая модель в основном опирается на нормальное распределение спроса с отсечением отрицательных значений. Для товаров с низким спросом, сезонными всплесками или редкими крупными заказами лучше рассмотреть:

- логнормальное распределение;
- Пуассон/negative binomial для штучного спроса;
- сценарную смесь для сезонности и промо;
- эмпирическое распределение по историческим продажам.

## Добавленные автоматические проверки

### Финансовые инварианты Black-Scholes

Добавлены тесты, которые фиксируют базовое поведение опциона:

- цена совпадает с эталоном Black-Scholes `10.4506` для `S=100, K=100, T=1, sigma=0.2, r=0.05`;
- цена растет при росте `S`;
- цена падает при росте `K`;
- цена растет при росте `sigma`;
- при `sigma = 0` цена равна `max(0, S - K * exp(-rT))`.

### Калибровка волатильности выручки

Добавлена функция `calibrateRevenueVolatilityFromSales`, которая собирает недельные бакеты выручки и оценивает годовую `sigma` по историческим продажам.

Проверки подтверждают:

- стабильная недельная выручка дает почти нулевую историческую волатильность;
- нестабильная недельная выручка дает более высокую `annualizedSigma`;
- волатильность из горизонта корректно переводится в годовую шкалу через `sqrt(T)`.

### Проверка stockout-искажений

Добавлен тест, что `computeWeeklyStatsAdjustedForStockouts` повышает оценку спроса, если продажи были ниже из-за отсутствия товара на складе. Это важно: без такой проверки модель может принять дефицит товара за низкий спрос и занизить `q*`.

### Удалены нечестные тесты

Удалены placeholder-проверки вида `expect(true).toBe(true)` и тайминговая проверка Monte Carlo через `Date.now()`. Такие тесты создавали видимость покрытия, но не проверяли математическое поведение.

## Рекомендации

### 1. Сделать строгую BS-оценку основным режимом для портфеля

Сценарная оценка через `strictBSMixtureOptionValue` лучше соответствует идее "опцион на будущую выручку", потому что сначала строится распределение выручки, а потом из него выводится `sigmaBS`.

### 2. Добавить калибровку по истории

Для каждого SKU желательно сохранять:

- фактический спрос;
- фактические продажи;
- stockout-события;
- цену закупки;
- цену продажи;
- логистические задержки;
- курс валюты.

На этих данных можно проверять, завышает или занижает модель `q*`.

### 3. Разделить `hold` и стоимость капитала

После исправления `r` отвечает за дисконтирование, а `hold` — за физическое хранение. Важно не включать стоимость капитала в `hold`, иначе появится повторный учет риска/процента уже на уровне пользовательских параметров.

### 4. Расширять тесты финансовых инвариантов

Уже добавлены базовые проверки:

- цена колла растет при росте `S`;
- цена колла падает при росте `K`;
- цена колла растет при росте `sigma`;
- при `sigma = 0` цена равна `max(0, S - K * exp(-rT))`;
- `calculateInventoryStrike` не зависит от `r`.

Следующий уровень — backtest на реальных SKU: сравнивать рекомендованный `q*` с фактической прибылью и stockout-событиями.

## Заключение

Математика проекта теперь согласованнее: Black-Scholes используется как оценка реального опциона на будущую выручку, `K` трактуется как стоимость исполнения складского решения, а `r` применяется только в дисконтировании. Главный следующий шаг — не менять саму идею Black-Scholes, а точнее калибровать распределение будущей выручки и волатильность по реальным данным.
